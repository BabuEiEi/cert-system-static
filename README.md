# ระบบสร้างและค้นหาเกียรติบัตรออนไลน์ (Certificate System)

PHP + TailwindCSS + SweetAlert2 + Firebase (Authentication + Firestore)
รองรับ 3 บทบาท: **Admin / Staff / User** — สร้างเกียรติบัตรเป็นไฟล์ PNG ได้ทีละคนหรือทั้งหมด (ZIP)

---

## 1) สถาปัตยกรรม: PHP ควรเชื่อม Firebase แบบไหนดี?

มี 2 ทางเลือกหลัก เปรียบเทียบดังนี้

| ประเด็น | แบบ A: Firebase JS SDK ฝั่งเบราว์เซอร์ (ที่ระบบนี้ใช้) | แบบ B: PHP เรียก Firebase ผ่าน REST/Admin SDK |
|---|---|---|
| ความยากในการติดตั้ง | ต่ำมาก — แค่ใส่ CDN | สูง — ต้องใช้ Composer, Service Account, cURL |
| โฮสติ้งที่รองรับ | Shared hosting ทั่วไป (โฮสต์โรงเรียน/หน่วยงาน) ได้ทันที | ต้องการสิทธิ์ติดตั้ง library และเก็บไฟล์ key อย่างปลอดภัย |
| ความปลอดภัย | ควบคุมด้วย **Firestore Security Rules** (ให้ไว้ในไฟล์ `firestore.rules`) | ควบคุมฝั่ง server ได้ละเอียดกว่า แต่ถ้า key หลุด = อันตรายมาก |
| การสร้าง PNG | ทำที่เบราว์เซอร์ด้วย Canvas → เร็ว ไม่กินทรัพยากรเซิร์ฟเวอร์ ไม่ต้องติดตั้ง GD/Imagick | ต้องติดตั้ง GD/Imagick + จัดการฟอนต์ไทยบนเซิร์ฟเวอร์ (ยุ่งยาก) |
| เหมาะกับ | ระบบเกียรติบัตรระดับหน่วยงาน/เขตพื้นที่/โรงเรียน | ระบบองค์กรใหญ่ที่ต้องมี backend logic ซับซ้อน |

**ข้อสรุปเชิงปฏิบัติ:** สำหรับโจทย์นี้ **แบบ A เหมาะสมที่สุด** — PHP ทำหน้าที่เป็นตัวประกอบหน้าเว็บ (templating เช่น `include` ส่วนหัวร่วมกัน) ส่วนข้อมูลและสิทธิ์ทั้งหมดคุยกับ Firebase โดยตรงจากเบราว์เซอร์ ภายใต้ Security Rules ที่บังคับบทบาทไว้แล้ว จุดแข็งสำคัญคือ **การสร้าง PNG ด้วย Canvas ฝั่ง client** ทำให้ฟอนต์ไทย (Sarabun ฯลฯ) แสดงผลถูกต้องโดยไม่ต้องติดตั้งอะไรบนเซิร์ฟเวอร์เลย

## 2) โครงสร้างไฟล์

```
cert-system/
├── index.html               ← หน้าค้นหา + ดาวน์โหลด (User)
├── login.html               ← เข้าสู่ระบบ (Admin/Staff)
├── dashboard.html           ← แผงควบคุม ออกแบบเกียรติบัตร
├── firestore.rules         ← กฎความปลอดภัย (วางใน Firebase Console)

└── assets/js/
    ├── firebase-config.js  ← ★ ใส่ค่าโปรเจกต์ Firebase ของคุณที่นี่
    ├── cert-engine.js      ← เครื่องมือวาดเกียรติบัตรลง Canvas → PNG
    ├── search.js           ← ตรรกะหน้าค้นหา
    └── dashboard.js        ← ตรรกะแผงควบคุม
```

## 3) โครงสร้างฐานข้อมูล (Firestore)

```
users/{uid}
  └─ { email, role: "admin" | "staff" | "user" }

activities/{activityId}
  ├─ { name, published, template{...}, createdBy, createdAt }
  └─ participants/{participantId}
       └─ { name, certNo }
```

- `template` เก็บการตั้งค่ารูปแบบทั้งหมด (สีพื้น, URL รูปพื้นหลัง, ฟอนต์, โลโก้, ลายเซ็น ฯลฯ)
- เกียรติบัตร **ไม่ถูกเก็บเป็นไฟล์** แต่ถูก "วาดใหม่" จาก template + ชื่อ ทุกครั้งที่ดาวน์โหลด → ประหยัดพื้นที่ 100% และแก้ template ย้อนหลังได้ทันที

## 4) สิทธิ์ตามบทบาท (บังคับด้วย firestore.rules)

| ความสามารถ | Admin | Staff | User |
|---|:-:|:-:|:-:|
| กำหนด/สร้าง/ลบกิจกรรม | ✅ | ❌ | ❌ |
| เผยแพร่กิจกรรมให้ค้นหาได้ | ✅ | ❌ | ❌ |
| ตั้งค่ารูปแบบ (สีพื้น, พื้นหลังจาก URL, ฟอนต์, โลโก้, ลายเซ็น) | ✅ | ✅ | ❌ |
| นำเข้ารายชื่อผู้อบรม | ✅ | ✅ | ❌ |
| สร้าง PNG (รายคน / ทั้งหมดเป็น ZIP) | ✅ | ✅ | ❌ |
| จัดการบทบาทผู้ใช้ | ✅ | ❌ | ❌ |
| ค้นหา + ดาวน์โหลดเกียรติบัตร | ✅ | ✅ | ✅ (ไม่ต้องล็อกอิน) |

## 5) ขั้นตอนติดตั้งแบบ Step-by-Step

### ขั้นที่ 1 — สร้างโปรเจกต์ Firebase
1. เข้า https://console.firebase.google.com → **Add project** → ตั้งชื่อ เช่น `cert-system`
2. ปิด Google Analytics ก็ได้ (ไม่จำเป็น) → Create

### ขั้นที่ 2 — เปิดใช้ Authentication
1. เมนู **Build → Authentication → Get started**
2. แท็บ **Sign-in method** → เปิด **Email/Password**
3. แท็บ **Users → Add user** สร้างบัญชีแรก เช่น `admin@school.ac.th` + รหัสผ่าน
4. **คัดลอก UID** ของบัญชีนี้ไว้ (จะใช้ในขั้นที่ 4)

### ขั้นที่ 3 — เปิดใช้ Firestore
1. เมนู **Build → Firestore Database → Create database**
2. เลือก location เช่น `asia-southeast1 (Singapore)` → เริ่มแบบ **Production mode**
3. ไปแท็บ **Rules** → ลบของเดิม → วางเนื้อหาจากไฟล์ `firestore.rules` → **Publish**

### ขั้นที่ 4 — สร้างเอกสาร role ของ Admin คนแรก
1. Firestore → **Start collection** → Collection ID: `users`
2. Document ID: **วาง UID จากขั้นที่ 2**
3. เพิ่ม field: `email` (string) = อีเมลแอดมิน, `role` (string) = `admin` → Save
   *(บัญชี Staff คนต่อ ๆ ไป สร้างใน Authentication แล้วให้ Admin ตั้งบทบาทผ่านปุ่ม "จัดการบทบาท" ในระบบ หรือเพิ่มเอกสารแบบเดียวกันโดยใส่ role = `staff`)*

### ขั้นที่ 5 — เชื่อมเว็บกับ Firebase
1. หน้า Project Overview → ไอคอน **</> (Web)** → ตั้งชื่อ app → Register
2. คัดลอกค่าใน `firebaseConfig` มาวางแทนที่ในไฟล์ `assets/js/firebase-config.js`

### ขั้นที่ 6 — นำขึ้นโฮสต์ PHP
1. อัปโหลดโฟลเดอร์ทั้งหมดขึ้น hosting ที่รัน PHP ได้ (ทดสอบในเครื่อง: `php -S localhost:8000` ในโฟลเดอร์โปรเจกต์)
2. เพิ่มโดเมนจริงของคุณใน **Authentication → Settings → Authorized domains** (localhost ถูกอนุญาตอยู่แล้ว)

### ขั้นที่ 7 — สร้าง Index สำหรับการค้นหาข้ามกิจกรรม
การค้นหาแบบ "ทุกกิจกรรม" ใช้ collection group query ครั้งแรกที่ค้นหา ระบบจะ error พร้อม **ลิงก์สร้าง index อัตโนมัติใน Console** — คลิกลิงก์นั้น → Create → รอ 1–2 นาที เสร็จ
(หรือสร้างเอง: Firestore → Indexes → Collection group `participants`, field `name` Ascending)

### ขั้นที่ 8 — ทดลองใช้งานครบวงจร
1. เข้า `login.html` ด้วยบัญชี admin → สร้างกิจกรรม
2. ตั้งค่ารูปแบบ (สี, พื้นหลัง URL, ฟอนต์, โลโก้, ลายเซ็น) → **พรีวิว** → **บันทึกรูปแบบ**
3. วางรายชื่อ (1 คน/บรรทัด) → **นำเข้ารายชื่อ** → กด **สร้าง PNG ทั้งหมด (ZIP)** ทดสอบ
4. ติ๊ก **เผยแพร่ให้ค้นหาได้**
5. เปิด `index.html` แบบไม่ล็อกอิน → เลือกกิจกรรม → พิมพ์ชื่อ → ดาวน์โหลด PNG ✅

## 6) ข้อควรรู้เรื่องรูปภาพจาก URL (สำคัญ)

Canvas จะ export PNG ได้ต่อเมื่อรูป (พื้นหลัง/โลโก้/ลายเซ็น) มาจากแหล่งที่อนุญาต **CORS**
แหล่งที่ใช้ได้แน่นอน: **Firebase Storage, Imgur, GitHub (raw), Cloudinary, Google Drive แบบลิงก์ direct**
ถ้ารูปโหลดไม่ขึ้น/ดาวน์โหลดไม่ได้ ให้ย้ายรูปไปฝากที่แหล่งข้างต้นก่อน

| ระยะ | งาน | เวลาโดยประมาณ |

| 2. ติดตั้งเว็บ | อัปโหลดไฟล์ขึ้นโฮสต์, ใส่ firebase-config, ทดสอบ login | 0.5 วัน |

| 4. ตั้งค่ากิจกรรมนำร่อง | Admin สร้างกิจกรรม 1 รายการ, Staff ออกแบบ + นำเข้ารายชื่อ, ทดสอบ ZIP | 1 วัน |

| 6. เปิดใช้จริง | เผยแพร่ทุกกิจกรรม + ประชาสัมพันธ์ลิงก์ | — |

## 8) แนวทางต่อยอด (เลือกทำภายหลัง)

- อัปโหลดรูปเข้า **Firebase Storage** โดยตรงจากหน้า dashboard (แทนการวาง URL)
- เพิ่ม **QR Code ตรวจสอบความแท้จริง** ที่มุมเกียรติบัตร ชี้ไปหน้า verify
- นำเข้ารายชื่อจากไฟล์ **CSV/Excel** (เพิ่ม SheetJS 1 บรรทัด)
- เก็บสถิติจำนวนดาวน์โหลดต่อกิจกรรม
